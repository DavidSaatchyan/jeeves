# T1.1 — OpenCode Multi-Agent Capabilities Research

**Date:** 2026-06-10
**Status:** Done
**Next:** T3.1 MCP Server, T3.3 Error Recovery

## Summary

OpenCode v1.17.0 does **not** have built-in multi-agent coordination (no Agent Teams, no SendMessage). Multi-agent workflow must be implemented as **Coordinator-Style** — swarm agent orchestrates sub-agents via `task` tool.

## Answers

### Q1: Agent Teams equivalent?
**No.** No built-in teams. Only subagent delegation via `task` tool.
- Source: `packages/core/src/agent/` — no teams concept

### Q2: SendMessage between subagents?
**No.** No inter-agent messaging infrastructure. Subagents cannot communicate directly — only through the orchestrator (swarm) or shared filesystem/MCP.

### Q3: Shared memory between tasks?
**No.** Sessions are fully isolated. Only filesystem is shared between task invocations.
- `task` tool spawns a new session per invocation
- Context must be passed explicitly via task prompt

### Q4: MCP tools in subagents?
**Yes.** Global MCP servers are registered and available to subagents, filtered by permissions declared in agent `.md` frontmatter.

### Q5: Plugin hooks (25+)
**Two plugin systems:**
1. **Internal (Effect-based):** Hooks at:
   - `Initialized`, `BeforeCommand`, `AfterCommand`, `ToolCall`, `ToolResult`, `MessageSent`, `MessageStreamed`
2. **User-facing JS plugins:** Hooks:
   - `config` — modify config on load
   - `tool.execute.before` — before tool execution
   - `tool.execute.after` — after tool execution
   - `chat.message` — when message is sent
   - `chat.params` — modify LLM params
   - `tool.definition` — modify tool definitions
   - `command.execute.before` — before command execution
   - `shell.env` — modify shell env
   - `permission.ask` — intercept permission prompts
   - `project.initialize.after` — after project init
   - `project.instructions` — modify project instructions
   - `session.initialize.after` — after session init
   - `session.restore.after` — after session restore
   - `session.error` — on session error

### Q6: `run_in_background`?
**Experimental.** Flag `OPENCODE_EXPERIMENTAL_BACKGROUND_SUBAGENTS=true` enables background execution. Not stable for production.

### Q7: Context passing between tasks?
Three mechanisms:
1. **Session chain** — tasks inherit parent context
2. **`task_id` reuse** — resume the same subagent session (continues with previous messages)
3. **Manual prompt crafting** — pass relevant context as task description

### Q8: Multi-agent coordination built-in?
**No.** Purely single-agent with subagent delegation. All coordination must be implemented at the application level.

## Recommendation

**Coordinator-Style (Swarm Orchestrator):**
- `swarm` agent receives user requests
- Calls `task` with `architect`/`implementer`/`reviewer`/etc.
- Passes context via task prompts
- Subagents use MCP for persistent memory (read/write decisions)
- No direct subagent-to-subagent communication

This aligns with OpenCode's architecture and avoids fighting the framework.

## Discovery Process

- Analyzed `packages/core/src/agent/` — no teams concept
- Analyzed `packages/core/src/task/` — `task` tool is sole delegation mechanism
- Analyzed `packages/core/src/plugin/` — cataloged Effect hooks
- Analyzed `packages/core/src/mcp/` — global registration, per-agent filtering
- Verified: no `SendMessage`, no `AgentTeams`, no `SharedMemory` in any source file
- Checked GitHub issues/discussions for multi-agent patterns — none found
